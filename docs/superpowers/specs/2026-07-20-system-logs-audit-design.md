# 日志管理页设计说明

## 背景

当前 `日志管理` 页面已具备基础筛选与表格能力，但内容较为空泛，难以支撑医疗系统中的审计追责场景。对于医疗后台，日志管理页面的核心价值不是简单展示“系统发生了什么”，而是快速回答以下问题：

- 谁在什么时间执行了什么操作
- 操作影响了哪个患者、档案、账号或系统对象
- 关键字段改了什么，改前和改后分别是什么
- 这次操作是否成功，是否属于高风险行为
- 操作来自什么设备、IP、入口页面或请求链路

因此，本次页面优化以“审计追责优先”为目标，围绕高可读表格、可追溯详情和医疗场景敏感字段展示进行设计。

## 设计目标

- 保持页面主体简洁，整页以日志表格为核心工作区
- 强化审计追责能力，突出“谁改了什么”
- 通过右侧 `Drawer` 展示完整事件详情，避免主表过宽
- 补充医疗系统风格的高质量 mock 数据，覆盖增删改查、登录审计、敏感访问和失败操作
- 保持当前后台风格一致，延续浅色、紧凑、专业的管理后台视觉语言

## 页面结构

页面采用“顶部筛选 + 全页表格 + 右侧详情抽屉”的结构：

1. 顶部保留筛选区
2. 中部整块区域为日志表格
3. 表格操作列提供“查看详情”
4. 点击“查看详情”后，从右侧打开 `Drawer`

该结构不采用常驻左右双栏，避免在默认状态下挤压表格宽度，保证列表检索效率。只有在用户主动查看某条日志时，才展示深度信息。

## 主表设计

### 表格定位

主表只承载“摘要信息”，用于快速扫描、筛选、定位目标日志。所有深度审计信息放入 `Drawer`。

### 建议字段

- 时间
- 日志类型
- 模块
- 操作人
- 操作对象
- 动作
- 改动摘要
- IP / 终端
- 结果
- 风险等级
- 操作

### 字段含义

- `时间`：事件发生时间，精确到秒
- `日志类型`：如登录审计、数据操作日志、系统日志
- `模块`：发生事件的业务模块，如档案管理、用户管理、预约管理
- `操作人`：账号或系统主体，如管理员账号、系统任务
- `操作对象`：被操作的数据对象，如患者档案、角色、用户、标签、中心
- `动作`：新增、编辑、删除、查看、导出、登录、重置密码等
- `改动摘要`：一句话概括本次改动，如“修改患者联系方式、随访标签”
- `IP / 终端`：来源 IP 与设备或浏览器摘要
- `结果`：成功或失败
- `风险等级`：低风险、中风险、高风险
- `操作`：查看详情

### 表格视觉规则

- 结果使用状态标签区分成功与失败
- 风险等级使用不同颜色标签区分风险级别
- 高风险日志行做轻量视觉强调，如浅红背景或弱红描边
- 敏感行为可在 `改动摘要` 或 `操作对象` 中进行关键词高亮
- `查看详情` 保持为单一文本按钮，避免操作列过重

## Drawer 详情设计

### 目标

右侧 `Drawer` 用于承载完整审计信息，使用户在不离开列表上下文的前提下完成追责和核查。

### 信息结构

#### 1. 事件摘要

- 日志编号
- 事件时间
- 日志类型
- 所属模块
- 操作结果
- 风险等级

#### 2. 操作主体

- 操作人账号
- 操作人姓名
- 角色
- 所属中心

#### 3. 操作对象

- 对象类型
- 对象标识
- 患者姓名或档案号（若适用）
- 业务归属信息

#### 4. 变更详情

以字段对比的方式展示：

- 字段名
- 修改前
- 修改后

该区域是本次设计的重点。对于编辑类操作，应尽可能展示真实字段变化；对于新增、删除、查看、导出等行为，则展示对应的事件说明。

#### 5. 访问上下文

- IP 地址
- 设备信息
- 浏览器
- 来源页面
- 请求方法
- 接口路径
- traceId 或 requestId

#### 6. 失败信息

仅在失败操作或异常日志中显示：

- 错误码
- 失败原因
- 拦截规则或校验结果

### 交互要求

- `Drawer` 从右侧滑出
- 默认宽度应适合展示字段对比信息
- 内容支持纵向滚动
- 关闭方式支持点击遮罩、右上角关闭按钮

## 数据范围设计

### 日志类型范围

- 登录审计
- 数据操作日志
- 系统日志

### 医疗场景优先内容

mock 数据应优先覆盖以下类型：

- 查看患者档案
- 编辑患者联系方式、标签、随访状态
- 删除预约记录
- 导出患者列表
- 重置账号密码
- 新增、编辑、删除角色
- 中心信息修改
- 登录成功与登录失败
- 敏感对象访问失败
- 系统任务执行异常

### 风险判断建议

- `高风险`：删除、导出、重置密码、敏感档案访问失败、批量操作
- `中风险`：编辑核心字段、角色权限变更、中心信息修改
- `低风险`：普通查看、登录成功、常规系统任务成功

## 筛选能力

保留现有筛选方式，并适度增强语义：

- 日志类型
- 模块
- 操作人
- 关键字
- 开始日期
- 结束日期

关键字建议同时匹配：

- 操作对象
- 改动摘要
- IP
- 患者档案号或对象 ID

## 架构与实现边界

### 本次实现包含

- 升级日志数据结构，支持更完整的审计字段
- 优化主表字段与视觉展示
- 增加 `查看详情`
- 接入右侧 `Drawer`
- 在 `Drawer` 中展示结构化详情
- 补充高质量医疗审计 mock 数据

### 本次实现不包含

- 真实后端接口接入
- 分页组件改造
- 导出文件下载能力
- 日志详情二次操作
- 多维统计图表

## 数据流设计

页面数据流保持轻量：

1. 初始化加载本地 mock 日志列表
2. 用户通过筛选条件过滤日志
3. 表格渲染过滤后的摘要数据
4. 用户点击某一行的“查看详情”
5. 将当前日志项设置为选中状态
6. 打开 `Drawer` 并渲染对应详情
7. 用户关闭 `Drawer` 后回到列表继续筛查

## 错误与边界处理

- 当筛选结果为空时，展示友好的空状态文案
- 当某些日志没有字段变更对比时，`Drawer` 中显示“无字段变更信息”
- 当日志来自系统任务而非人工账号时，操作主体显示为 `system`
- 当失败日志没有完整错误码时，至少展示失败原因摘要
- 当操作对象不是患者档案时，避免强行渲染患者字段

## 测试与验证建议

本次以静态页面和 mock 交互为主，验证重点如下：

- 表格字段显示是否完整、对齐是否稳定
- 筛选逻辑是否仍然正确
- 点击“查看详情”是否能打开正确的 `Drawer`
- `Drawer` 内容结构是否清晰，长内容是否可滚动
- 不同日志类型在详情中的展示是否合理
- 失败日志与高风险日志是否有足够辨识度

## 推荐落地方案

采用“全页表格主导 + 操作列查看详情 + 右侧 Drawer 展示完整审计信息”的方案。

该方案兼顾以下几点：

- 符合后台系统常见使用习惯
- 检索效率高，适合高频筛查
- 主表信息密度高但不失控
- 深度审计信息集中在详情层，不干扰主表浏览
- 非常适合医疗场景下的追责需求
