# 小程序问诊页免费预约 Agent 设计

**背景**

当前小程序端 `问诊` 页底部已经有 `免费预约`、`检查报告`、`就诊记录` 等快捷入口，但 `免费预约` 仍停留在静态按钮层，没有承接用户“看完医生后立即发起挂号”的连续动作。用户希望这里不是跳出式表单，而是像 Agent 一样，在当前问诊上下文中主动理解意图、逐步追问并帮助完成预约。

本次需求聚焦在 `免费预约` 点击后的悬浮交互：保留问诊页当前布局和底部输入区，在输入区上方原地浮出一问一答式卡片，让 Eye 宝完成“确认医生 -> 选择挂号方式 -> 选择未来一周号源”的轻量预约引导。

**目标**

- 点击 `免费预约` 后，不跳走页面，而是在当前 `问诊` 页原地浮出 Agent 对话卡片
- 默认使用本地 Mock 医生数据，首版固定展示用户上次看过的医生为 `上海眼病防治中心 徐蔚主任`
- 第一张卡片以 Agent 口吻确认是否直接帮助挂号
- 第一张卡片提供 2 个动作：
  - `去预约页面挂号`
  - `直接挂号`
- 点击 `直接挂号` 后，在当前输入区上方继续浮出第二张卡片，展示徐主任未来一周可预约号源
- 用户点击某个号源后，完成一次“已为您预约”的成功态反馈，形成完整闭环
- 整体视觉和问诊页现有浅蓝、白色、悬浮感风格一致，不改造成表单页或重弹窗

**确认方案**

用户已确认采用 `A 叠层对话卡片` 方案：

- 卡片在问诊页底部输入区上方原地浮出
- 第一张卡片保留，作为上下文历史，不被第二张卡片替换
- 点击 `直接挂号` 后，第二张卡片叠层出现，形成连续的“一问一答”效果
- 第二张卡片中的号源采用可点击胶囊或小卡片展示
- 所有流程均基于本地 Mock 数据完成，不接真实接口

**页面结构**

1. 问诊页主体保持不变

- 保留顶部 `Eye宝` 吉祥物区域
- 保留“你可以这样问我”快捷问题列表
- 保留底部固定输入区和快捷胶囊区
- `免费预约` 仍位于底部快捷胶囊首位

2. Agent 预约浮层区

- 位于底部输入区上方，悬浮在问诊内容之上
- 仅在用户点击 `免费预约` 后出现
- 浮层最多展示 2 张连续对话卡片和 1 段成功反馈，不扩展成长聊天列表
- 浮层需与底部输入区有明确层次间距，避免遮挡输入框

3. 第一张确认卡片

- 发送者为 `Eye宝`
- 文案示例：
  - `您上次看的是上海眼病防治中心徐蔚主任，是否直接帮您挂号？`
- 卡片底部包含两个操作按钮：
  - `去预约页面挂号`
  - `直接挂号`
- `去预约页面挂号` 为次级按钮
- `直接挂号` 为主按钮

4. 第二张号源选择卡片

- 仅在点击 `直接挂号` 后出现
- 仍由 `Eye宝` 发起
- 文案示例：
  - `徐主任未来一周这几天有号，帮您筛好了。`
- 下方展示未来一周号源列表，推荐使用 3 到 4 个可点击时间胶囊
- 号源信息至少包含：
  - 星期
  - 日期
  - 时段或具体时间
- 示例：
  - `周二 07/22 09:30`
  - `周四 07/24 14:00`
  - `周六 07/26 10:00`

5. 成功反馈态

- 点击任一号源后，在卡片区追加一条简短成功反馈
- 文案示例：
  - `已为您锁定徐蔚主任周四 07/24 14:00 的号源`
  - `稍后可在预约记录中查看`
- 成功态可以附带 1 个轻量操作，如：
  - `查看预约`
- 首版该按钮允许保留展示态，不要求跳转到真实页面

**交互流程**

1. 进入流程

- 用户点击底部快捷胶囊 `免费预约`
- 页面不跳转、不遮罩全屏
- 第一张确认卡片从底部输入区上方淡入上浮出现

2. 选择挂号方式

- 点击 `去预约页面挂号`
  - 不展开第二张卡片
  - 触发占位行为即可，例如弹出提示或预留回调
- 点击 `直接挂号`
  - 保留第一张卡片
  - 在其下方继续浮出第二张号源选择卡片

3. 选择号源

- 用户点击某个未来一周号源
- 当前选中的时间胶囊进入选中态
- 卡片区追加成功反馈
- 为避免状态复杂化，首版点击不同号源时可直接覆盖上一次选择结果

4. 重复打开逻辑

- 若用户已完成一次预约，再次点击 `免费预约`
  - 默认重置为第一张确认卡片初始态
- 不要求保留历史预约步骤到下一次点击

**状态设计**

1. 初始态

- 不显示 Agent 预约卡片
- 问诊页保持当前默认布局

2. 第一步进行中

- 显示第一张确认卡片
- 第二张号源卡片尚未出现

3. 第二步进行中

- 显示第一张确认卡片
- 显示第二张号源选择卡片
- 尚未出现成功反馈

4. 已完成态

- 显示第一张确认卡片
- 显示第二张号源选择卡片
- 显示成功反馈或预约完成提示

**数据策略**

本次仅使用本地 Mock 数据，不接真实接口或全局状态。

建议在 `ConsultModule.tsx` 内维护轻量数据结构：

```ts
type BookingAgentDoctor = {
  id: string;
  name: string;
  hospital: string;
  title: string;
};

type BookingAgentSlot = {
  id: string;
  label: string;
  date: string;
  time: string;
};
```

Mock 约定：

- 医生固定为：
  - `徐蔚主任`
  - `上海眼病防治中心`
- 未来一周号源固定提供 3 到 4 个
- 所有数据用于原型展示，不考虑接口失败、库存变化、并发占号等真实业务问题

**视觉设计**

- 整体风格延续 `ConsultModule` 已有的浅蓝、白色、圆角、轻阴影体系
- 第一张卡片建议使用浅蓝底或白底蓝描边，突出来自 Agent 的主动建议
- 第二张卡片使用白底悬浮卡，以与第一张卡片形成层级差异
- 两张卡片之间保留轻微错层或上浮动画，强调“对话推进”
- 时间号源使用胶囊按钮，默认浅蓝底，选中后切换为高饱和主蓝
- 动效以轻量为主：
  - 上浮
  - 淡入
  - 按钮按压反馈
- 不使用全屏抽屉、系统弹窗、厚重遮罩，以免破坏当前问诊页轻巧感

**文案原则**

- 语气要像 Eye宝 在“帮用户做决定”，而不是让用户自己填表
- 文案尽量短促、明确、带服务感
- 推荐使用：
  - `您上次看的是...`
  - `是否直接帮您挂号？`
  - `未来一周这几天有号，帮您筛好了`
  - `已为您锁定...`

**影响文件**

- 修改 `src/pages/miniprogram/components/ConsultModule.tsx`

如实现时需要把状态上提给外层容器，再按需小范围调整：

- 可选修改 `src/pages/miniprogram/index.tsx`

**不做事项**

- 不接真实“最近浏览医生”数据
- 不接真实预约接口
- 不跳转真实预约结果页
- 不引入聊天模型或后端对话能力
- 不做长链路多轮对话，只覆盖本次预约 2 步主路径
- 不处理真实号源售罄、并发冲突、异常兜底

**验证方式**

- 视觉检查：
  - 点击 `免费预约` 后卡片出现在底部输入区上方
  - 点击 `直接挂号` 后第二张卡片继续浮出
  - 时间胶囊可选中，并出现成功反馈
- 交互检查：
  - 点击 `去预约页面挂号` 有明确占位反馈
  - 再次点击 `免费预约` 时流程可回到初始态
- 代码验证：
  - `ConsultModule.tsx` 无新增 TypeScript / JSX 诊断错误
  - 原型手机壳内无明显溢出、遮挡或滚动冲突
